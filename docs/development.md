## Development commands

Toolchain is pinned via `mise.toml` (rust / node / pnpm / cargo-nextest). The package manager is **pnpm**; run all commands from the repo root (the Cargo workspace lives there, not under `src-tauri/`).

```bash
# Rust â€” pure core (the TDD battleground; -p keeps Tauri out)
cargo nextest run -p simple-archiver-core   # run tests (use nextest, not `cargo test`)
cargo nextest run -p simple-archiver-core --release  # release-only tests for #[cfg(not(debug_assertions))] paths (debug run skips them; see testing policy)
cargo clippy -p simple-archiver-core --all-targets -- -D warnings  # lint, zero warnings
cargo fmt                                    # format (whole workspace)
cargo llvm-cov nextest -p simple-archiver-core --lcov --output-path lcov.info  # coverage (CI uploads to Codecov)

# loom concurrency verification (target tests only; kept separate from normal runs)
RUSTFLAGS="--cfg loom" cargo clippy -p simple-archiver-core --features loom -- -D warnings  # loom-clippy (CI's loom job runs this too)
RUSTFLAGS="--cfg loom" cargo nextest run -p simple-archiver-core --features loom

# Rust â€” presentation crate (Tauri commands, IPC layer; requires the Tauri toolchain)
cargo clippy -p simple-archiver --all-targets -- -D warnings
cargo nextest run -p simple-archiver          # also regenerates ts-rs bindings in src/bindings/

# Frontend
pnpm format                # oxfmt: format all frontend files (write mode)
pnpm fmt:check             # oxfmt: check formatting, no writes (CI read-only gate)
pnpm lint                  # oxlint: lint frontend files
pnpm lint:ci               # oxlint: lint with --deny-warnings (CI gate)
pnpm check                 # oxfmt write + oxlint --fix (run before committing)
pnpm test                  # Vitest one-shot (= vitest run)
pnpm test:watch            # Vitest watch mode
pnpm run test:coverage     # Vitest coverage -> coverage/lcov.info
pnpm knip                  # unused files / deps / exports gate
pnpm build                 # tsc + vite build

# App run / build
pnpm tauri dev
pnpm tauri build
```

> In non-TTY/CI contexts, invoke pnpm as `mise exec -- pnpm ...`: the mise pnpm shim swallows stdout/exit code in non-TTY, so a success can look like a failure.

## Testing policy (TDD)

This project is designed around TDD. **Write tests before implementation.**

- **Removal / deletion refactors are also TDD â€” adjust tests first, then delete (HARNESS â€” PR-12b).** When deleting a field/accessor/concept, edit the tests to the new expected end-state *first* â€” while the symbol still exists â€” and confirm green; only then delete the field/accessor; then re-run and confirm the meaningful invariants stay green. This keeps the safety net live across the removal instead of deleting code and tests in one untested leap. **Sweep EVERY textual reference, not just code and tests:** grep the whole tree for the removed name and update PRODUCTION doc comments too, not only call sites. (PR-12b removed `ArchiveTask::progress` â€” a dead field always equal to `TaskProgress::zero()` with no write path; the plan's file-by-file enumeration missed 3 stale production doc comments in `archive_job.rs` that review caught. A reference list assembled by hand is not a substitute for a final `rg <name>` across `crates/ src-tauri/ docs/ .claude/`.)
- **Domain**: pure unit tests. Cover `NamingRule` resolution, `ArchiveJob.plan`/`reorder` invariants, and name-uniqueness boundary cases. For any **state machine** (e.g. `TaskStatus`), test every `(state Ă— event)` pair: legal transitions assert the resulting state, illegal ones assert the typed error, and terminal states reject all events.
  - **HARNESS â€” `debug_assert` + release-clamp invariants need cfg-split tests AND a `--release` gate.** A value object that asserts an invariant in debug and clamps it in release (e.g. `TaskProgress::new`; see conventions.md "debug_assert + release clamp for caller-bug invariants") needs two cfg-gated tests: `#[cfg(debug_assertions)] #[should_panic(expected = "â€¦")]` for the loud debug path, and `#[cfg(not(debug_assertions))]` asserting the clamped value for the release path. **Critical:** the default `cargo nextest run` is a debug build and never executes the `#[cfg(not(debug_assertions))]` test, so the verification gate MUST also include a `cargo nextest run --release` pass to exercise release-only tests â€” otherwise the clamp path is silently never run.
- **Application**: verify parallelism and error tallying **deterministically**. PR-5a uses **hand-written fakes** (`FakeArchiver` / `FixedClock` / `RecordingSink`), not `mockall`: the evolved `Archiver` returns `impl Future + Send` (RPITIT), which `mockall` cannot mock cleanly, and a purpose-built fake gives the control the timing assertions need â€” a `tokio::sync::Barrier(N)` + `timeout` proves N workers run concurrently, and the **cap** is proven by running *more* tasks than the limit and asserting peak liveness `== N` (sampled at the barrier rendezvous). `mockall` stays declared in dev-dependencies for a future PR that mocks the ports; PR-5b's cancel-path tests use the hand-written `FakeArchiver`, not mockall. `Clock` is faked with a fixed `Instant` so `elapsed` is deterministic.
- **Concurrency**: verify the **single-owner aggregator** (fed by an internal mpsc channel; single writer, no shared lock) with `loom`. `loom` is a normal `optional` dependency on `simple-archiver-core` gated behind the `loom` feature (Cargo cannot mark dev-dependencies optional). It is off by default, so normal builds never compile it; the verification run above flips the feature on. **loom is incompatible with `tokio::fs`**: under `RUSTFLAGS="--cfg loom"` tokio disables its IO modules (`fs`, net, â€¦) because loom only model-checks concurrency primitives. Pure-domain modules (no IO/async) need **no** `#[cfg(not(loom))]` guard and must keep compiling and passing under the `loom` feature â€” only modules that use `tokio::fs` or other IO require the guard. So IO adapters that use `tokio::fs` (e.g. `ZipArchiver`) must be excluded from loom builds via `#[cfg(not(loom))]` (on the `infrastructure::zip_archiver` module declaration and on its integration test file), and the bare `cfg(loom)` name must be declared in `crates/core/Cargo.toml` under `[lints.rust] unexpected_cfgs = { check-cfg = ['cfg(loom)'] }` so `cargo clippy -D warnings` stays clean. PR-5a builds the engine but adds **no** loom tests; **PR-5b adds the loom nucleus** (`application/loom_nucleus.rs`, gated `#[cfg(loom)]`) which drives the real `Aggregator`/`WorkerMsg`/`ArchiveJob` under loom primitives â€” three concurrent loom-thread workers send terminal/progress messages over a loom mpsc channel into the single-writer aggregator. Under every loom interleaving it verifies that no message is lost and the summary partitions every task into exactly one of succeeded/cancelled/failed. This checks the **concurrency model** (single-writer aggregation), NOT the tokio runtime, and NOT `CancellationToken` propagation (that signal path is covered by the regular tokio cancel-path tests). Note: loom 0.7 models message count, not sender-drop/channel closure, so the production workerâ†’channel-closeâ†’drain ordering is covered by the regular tokio tests, not loom. **Harness consequence:** because loom's `recv()` blocks forever on an empty channel (no "all senders dropped" signal), a `while let Ok(_) = rx.recv()` drain-to-closure *deadlocks* under loom â€” the loom model must instead drain an **exact, known message count** derived from named constants that mirror what the workers send (e.g. `SUCCESS_MSGS + CANCEL_MSGS + FAILURE_MSGS`). Over-draining hangs; under-draining leaves a non-terminal task. Keep the count constant-derived (not a magic literal) so it stays honest as the worker set changes. Two modules are gated `#[cfg(not(loom))]` â€” `application::run_archive_job` (uses the tokio runtime: `rt`/`sync`/`spawn`/`mpsc`/`Semaphore`) and `infrastructure::zip_archiver` (`tokio::fs`) â€” while every other new module (`progress`, `compress_context`, `progress_aggregator`, `ports`, `system_clock`) is pure/std and must keep compiling and passing under the `loom` feature. So the loom build now compiles down to the pure domain **plus** those std/pure application modules. **Generalised guard rule (harness â€” PR8 build break):** ANY `#[cfg(not(loom))]`-gated infrastructure module requires `#![cfg(not(loom))]` at the top of EVERY integration test file (`crates/core/tests/*.rs`) that references it â€” not only `tokio::fs` adapters. `infrastructure::unrar_extractor` is gated `not(loom)` because it uses `tokio::task::spawn_blocking` over the synchronous `unrar` C API (not `tokio::fs`), and its smoke/pipeline integration tests (`unrar_extractor_smoke.rs`, `rar_pipeline_smoke.rs`) each need the `#![cfg(not(loom))]` inner attribute or the whole-package `--features loom` nextest run fails to **compile** â€” the test file references a module compiled out under loom.
- **Infrastructure**: narrow integration tests against small real fixtures (small folder, small .rar). Few and slow, so isolate them with a marker. **RAR fixture (harness â€” PR8):** The committed fixture `crates/core/tests/fixtures/sample.rar` is a **real (user-provided) RAR5** archive containing a single `hello world.txt` = `"hello world"` (11 bytes, no trailing newline; 90 bytes total). Both RAR4 and RAR5 are handled transparently by the `unrar` adapter. The final correctness oracle is the Rust `unrar`-backed integration test, not just `7zz`. **Fallback â€” if you cannot obtain a real `.rar` (e.g. WinRAR/`rar` is unavailable on your machine or in CI):** RAR is a proprietary format â€” there is no open-source tool that *creates* a `.rar` archive (7-Zip/`7zz` can list and extract rar but cannot author one; only WinRAR/`rar` can). You can hand-craft a minimal RAR4 "stored" (uncompressed, METHOD 0x30) archive in a script: write the marker block `52 61 72 21 1A 07 00`, a MAIN header (type 0x73), then a FILE header (type 0x74) whose HEAD_CRC is the low 16 bits of `crc32` over the header bytes and whose FILE_CRC32 is `crc32` of the raw file data (Python `zlib.crc32`), followed by the stored bytes; verify with `7zz l` / `7zz x`.
  - **A test asserting absence must first prove presence (no vacuous side-effect tests).** A cleanup/cancel test that fires its trigger at the *first* checkpoint runs **before** the side-effect happens (e.g. the output file is not yet created), so an "artifact was removed" assertion passes vacuously and never exercises the real cleanup path. Fire the trigger *after* the side-effect and prove it occurred first â€” e.g. assert the progress reporter was called â‰Ą2 times â‡’ â‰Ą1 entry was written â‡’ the dest file genuinely existed â€” *then* assert it was cleaned up (`ZipArchiver`'s `cancels_after_a_write_removes_the_partial_output` is the canonical shape). **Harness extension (PR10) â€” nondeterministic mid-run cancellation:** for an integration-level cancel test where the exact cancel point is nondeterministic, use enough source entries (PR10 uses 256) that the cancel token is observed before the job completes. If the test ever flakes (job finishes before cancel lands), **raise** the entry count â€” never lower it. Use a progress-driven emitter that records a `fired` flag on the first `overall.bytes_done > 0` event; this flag both triggers the cancel and self-guards against a false pass: if the job somehow completed before cancel was observed, the subsequent `cancelled`-count assertion fails loudly rather than passing silently. The fully-deterministic proof remains the unit test; the integration test complements it at the full-pipeline level without replacing it.
- **Presentation (Tauri commands)**: a `#[tauri::command] pub fn` is an ordinary Rust function; its `Result<_, String>` mapping can be asserted in a plain `#[cfg(test)]` unit test **without** constructing an `App` or `Window`. Integration-level coverage (command seam: argument names `src`/`out`, `ArchiveError`â†’`String` IPC mapping) can be added via `src-tauri/tests/*.rs` â€” the macro does not consume the original fn. Requires: the `presentation` module is `pub`, the lib crate is `simple_archiver_lib` with `[lib] crate-type` including `"rlib"`, and dev-deps `tokio` (macros, rt-multi-thread) + `zip` + `tempfile`.
  - **HARNESS â€” IPC/wire layer testing** (origin: PR6). Use three complementary layers:
    1. **Serde-shape tests** (unit): assert `serde_json::to_value(&dto)` equals the expected camelCase JSON structure and that snake_case keys are absent. These pin the wire contract without a running Tauri app.
    2. **`RecordingEmitter` test double** (unit): implement the `ProgressEmitter` trait with a `Mutex<Vec<ProgressEvent>>` that records every emitted event. Use it in unit tests that need to verify event ordering or field mapping without an `AppHandle`.
    3. **`run_job_inner` integration tests** (`src-tauri/tests/run_job_command.rs`): drive the IPC-free core with a real `ZipArchiver` + `SystemClock` + temp dirs. Happy path: assert `summary.succeeded.len() == N`, that at least one progress event was emitted with `overall.bytes_done > 0`, and that output zips exist on disk. Pre-cancelled path: assert the cancelled **count** (not exact ids â€” thread-pool order is nondeterministic) and that no output zips were written.
       **PR10 integration test additions:**
       - **Deterministic mixed rar+folder happy path**: a `run_job_inner` test that feeds a job containing both a `.rar` source and a plain folder, then asserts `summary.succeeded.len()` equals the total task count and that the expected output zip files exist on disk. This exercises the full `FormatRegistry` dispatch (rar â†’ extract â†’ compress AND folder â†’ compress) end-to-end through the IPC seam.
       - **Many-file mid-run cancellation test**: a `run_job_inner` test that builds a job with many source files, fires a cancellation token mid-run (after the first progress event with `overall.bytes_done > 0` proves at least one entry was written), and asserts that no partial zip remains on disk. The high file count makes the mid-run cancel land practically certain â€” with few entries the job often finishes before the cancel token is observed. The core unit-level, fully-deterministic proof that a partial zip is cleaned up on cancel remains `zip_archiver::tests::cancels_after_a_write_removes_the_partial_output`; the integration test complements it at the full-pipeline level.
  - **Testing constraint â€” `TaskId` visibility**: `TaskId::new` is `pub(crate)`, so tests in the separate `src-tauri` crate cannot fabricate a `TaskId`. Cover `From<JobSummary>`/`From<&JobProgress>` mappings with empty id collections and rely on serde-shape tests for field-level contract coverage. Tests within `simple-archiver-core` that need real `TaskId` values obtain them via `ArchiveJob::plan(items, rule, out_dir).unwrap().tasks()[i].id()` â€” the aggregate root is the only sanctioned constructor path.
- **Frontend**: Vitest on jsdom + Testing Library for preview computation, reordering, and progress rendering, including event-payload contract tests. Mock `@tauri-apps/api/core` (`invoke`), `@tauri-apps/api/event` (`listen`), `@tauri-apps/api/webview` (`getCurrentWebview().onDragDropEvent`), and `@tauri-apps/plugin-dialog` (`open`).
  - **HARNESS â€” `pnpm test` does NOT type-check; `pnpm build` is the real type gate** (origin: PR7). Vitest transpiles per-file via esbuild and never runs `tsc`, so a vitest-green run can still fail the type checker. `tsconfig.json` has `include: ["src"]` with **no** test exclusion, so `pnpm build` (`tsc && vite build`) type-checks `*.test.tsx` too. **Always run `pnpm build` before claiming a frontend change is done** â€” vitest-green â‰  build-green. Concrete gotchas this surfaced: `screen.getByRole("button", â€¦)` returns `HTMLElement`, so reading `.disabled` needs `as HTMLButtonElement`; `JobSummaryDto.failed` is `FailedTaskDto[]` (`{ taskId, reason }`), not `number[]`; replacing a non-null assertion `x!` with `x?.()` can re-narrow a callback-captured variable to a non-callable `null`/`never` â€” use a typed local/cast instead. **HARNESS (PR9) â€” adding a required field to a DTO/binding type requires backfilling ALL existing test fixtures.** When a required field is added to a generated binding (`ProgressEvent.overallEtaMs`, `TaskProgressDto.etaMs`), Vitest stays green because esbuild never type-checks â€” but `pnpm tsc --noEmit` (or `pnpm build`) fails immediately on every test file that constructs a literal of that type without the new field. Rule: any change to required DTO or binding fields must be followed by `pnpm tsc --noEmit`; every existing fixture constructing that literal must be backfilled before committing.
  - **Components are tested against the REAL zustand store, not a mocked module.** Call `resetJobStore()` in `beforeEach` (it relies on zustand v5 `getInitialState()` + replace-mode `setState(getInitialState(), true)`, which restores the real actions too); arrange state with `useJobStore.setState({...})`; inject spy actions with `useJobStore.setState({ someAction: vi.fn() })` (components select actions from the store, so the spy is picked up).
  - **Deterministically test async ordering** (e.g. a `running` flag that must be true before an await resolves, or a generation/staleness guard) by having the mocked command wrappers return manually-controlled promises and resolving them by hand in the intended order.
  - **Native DOM assertions only** â€” jest-dom is intentionally not installed. Use `el.className`, `classList.contains(â€¦)`, `.textContent`, `.disabled`, etc. Never `toBeInTheDocument` / `toHaveClass`.
  - **Testing an `aria-disabled` (not natively `disabled`) action requires both activation paths plus the focusability contract (PR11 follow-up).** The button stays focusable and the only block is a JS handler guard, so assert: (1) a mouse `click` while disabled does NOT call the action; (2) a keyboard `{Enter}` (via `userEvent.keyboard` after `.focus()`) while disabled also does NOT; (3) the button stays focusable â€” `(btn as HTMLButtonElement).disabled === false` while disabled; (4) the reason is wired for AT â€” resolve `aria-describedby` to its node in the live DOM (a `runReasonText`-style helper) and assert the text plus the `title` string for every disabled reason. Covering only the click path leaves the keyboard guard â€” the whole point of `aria-disabled` â€” untested.
  - **CSS variables are not resolvable in jsdom.** Do not assert computed token values (colors, radii) in unit tests â€” those can only be verified visually or in a browser. Confirm token wiring compiles correctly via `pnpm build`.
  - Shared test helpers and stubs live in `src/test/` (e.g. `setup.ts` for cleanup, `stub-match-media.ts`). The knip entry point covers `src/components/ui/**` so shadcn primitives are not flagged unused.
  - `@testing-library/user-event` treats `{` as a special key sequence â€” type a literal brace as `{{` (e.g. `img_{{n:03}` produces `img_{n:03}`).
  - Combining `vi.useFakeTimers()` with `userEvent` can deadlock in jsdom. For debounce-timing tests, drive input with `fireEvent.change` + `vi.advanceTimersByTimeAsync`, and confirm the test fails when the debounce is removed as a correctness check.

## Coverage (Codecov)

Coverage is reported to Codecov **informationally** â€” it never blocks a PR. The
ubuntu `core` and `frontend` CI jobs upload lcov to Codecov under flags `rust`
and `frontend` (auth via the `CODECOV_TOKEN` repository secret). The
mac/windows `app` job runs tests but does not upload coverage. The repository must be
activated on codecov.io for reports to appear. Coverage config lives in
`codecov.yml`; local lcov artifacts (`lcov.info`, `coverage/`) are git-ignored.

## Commit / PR rules

- Target **one PR â‰¤ 1000 lines**, with the walking skeleton (folder â†’ zip e2e) going first. - Stack PRs in order of high-impact Ă— low-effort (follow the PR1â€“PR10 split and dependency graph in the design).
- Each PR has acceptance criteria in the design; do not mix in out-of-scope items.
- Before merge, `cargo clippy` / `cargo fmt` / `cargo nextest run` / `pnpm fmt:check` / `pnpm lint:ci` / `pnpm test` / `pnpm build` must all be green. `pnpm build` (= `tsc && vite build`) is the load-bearing type gate â€” `pnpm test` does not type-check (see Frontend testing policy). CI builds on both Mac and Windows. **If any DTO or `SourceKind` variant changed**, also run the binding export test (`cargo test export_typescript_bindings --manifest-path src-tauri/Cargo.toml`) followed by `pnpm format`, then verify `git diff src/bindings` is limited to the intended change before committing (see ts-rs binding workflow in Build / scaffold notes).
- **HARNESS â€” scoped runs do NOT substitute for the full pre-commit gates.** `vitest run <file>` and a filtered `cargo nextest run -p simple-archiver <test-name>` do not exercise the whole-project gates. The following are only caught by the full gates and **must** be run before committing: `pnpm fmt:check` (oxfmt drift), `pnpm lint:ci` (oxlint `--deny-warnings` â€” e.g. the `jsx-a11y/prefer-tag-over-role` rule fires only here), `pnpm build` (tsc type-check â€” Vitest does not type-check), `pnpm knip` (unused exports/files), and `cargo clippy --all-targets`. Parallel per-file test passes while the full gate catches a lint or type error is the failure mode this rule prevents.
- Never run `git commit` / `git push` until the user explicitly asks.

## Build / scaffold notes (learnings)

- **`bitflags` pinned to `=2.8.0` (src-tauri).** dispatch2 (transitive via taoâ†’wryâ†’tauri) emits a large `bitflags!` block; bitflags 2.9+ expands it recursively and overflows the default `recursion_limit = 128`. 2.8.0 is non-recursive. Remove the pin once dispatch2 sets `#![recursion_limit]` or bitflags fixes the recursion. (See the comment in `src-tauri/Cargo.toml`.)
- **oxlint/oxfmt path scope (harness):** all lint and format scripts target `src vite.config.ts vitest.config.ts` â€” NEVER the repo root `.`. The repo root holds the Rust workspace (`Cargo.toml`, `src-tauri/`, `crates/`) and CI configs that must not be reformatted by the frontend formatter. Keep the explicit frontend path list when editing or adding scripts in `package.json`.
- **pnpm build-script approval lives in `pnpm-workspace.yaml`.** `onlyBuiltDependencies: [esbuild]` must sit there: pnpm 11 ignores `pnpm.onlyBuiltDependencies` in `package.json` and treats `allowBuilds` as a no-op. Without approval, esbuild's postinstall is skipped and `vite build` fails under CI's `--frozen-lockfile`.
- **Tailwind v4 + shadcn needs explicit theme glue.** The global CSS requires a `@theme inline { --color-*: var(--*) }` block plus `@custom-variant dark`. v4 does not auto-generate `bg-primary` / `border-ring` etc. from bare `:root` variables; without the glue the build passes but shadcn components render unstyled.
- **Follow-up (PR10 bundling):** the Tauri bundle identifier `com.simplearchiver.app` ends in `.app` and triggers a warning; change it before the bundling PR. This finishing-touches PR (PR10) did **not** rename it â€” renaming was out of scope per Issue #10 and remains a follow-up item for the dedicated bundling PR.
- **logos + LALRPOP build wiring.** The `core` crate uses a `build.rs` that calls `lalrpop::process_root()` to codegen the parser from `src/domain/template.lalrpop`. The generated file is referenced with `lalrpop_util::lalrpop_mod!(name, "/domain/template.rs")` â€” the path must mirror the `.lalrpop` file location. `lalrpop` (build-dep) and `lalrpop-util` (dep) **must be pinned to the same version** (0.20.x); version skew causes silent codegen mismatches. logos provides the lexer; its `SpannedIter` is adapted into `(start, token, end)` triples for LALRPOP's external-lexer mode.
- **A `#[cfg]`-gated test suite needs its own CI job or it silently bit-rots.** A `#[cfg(loom)]`-gated suite is compiled *out* of every normal CI job, so without a dedicated `RUSTFLAGS="--cfg loom" cargo nextest --features loom` job (plus a matching loom-clippy step) it never compiles or runs and rots unnoticed. The `loom` CI job exists for exactly this. Related tooling note: once a cfg-gated module contains `use loom::â€¦`, cargo-machete (a text scan) counts the optional dep as *used*, so it can be removed from the machete `ignored` list (loom was; `mockall` stays ignored while still unreferenced).
- **loom module-gating rule (harness â€” PR9): a module reachable ONLY from `#[cfg(not(loom))]` code must itself be gated `#[cfg(not(loom))]`.** A module that is only ever imported or called from code already gated `not(loom)` (e.g. `eta_estimator`, used only by `run_archive_job`) will produce dead-code warnings under `--cfg loom` if left always-compiled. Gate it to match its callers. Conversely, modules whose public interface changes shape (e.g. `JobProgress` adding an `overall_eta` field) must still compile under loom â€” keep the loom nucleus and any always-compiled pure modules consistent with the updated struct layout.
- **`cargo fmt` is a separate CI gate â€” not enforced by clippy (harness â€” PR8).** `cargo clippy -D warnings` passes on unformatted code. CI's `core` and `app` jobs run `cargo fmt --check` as a **separate** step, so code can be clippy-clean yet fail CI on formatting. Harness: run `cargo fmt` (or `cargo fmt --check`) on every commit that adds or edits Rust, not only at the end of a PR â€” PR8 missed this per-task and needed a trailing `style: apply cargo fmt` cleanup commit.
- **clippy `-D warnings` gotchas (CI gate).** Two patterns tripped this PR:
  - Use `!(1..=N).contains(&x)` instead of `x < 1 || x > N`; the latter triggers `clippy::manual_range_contains`.
  - A `pub` / `pub(crate)` item is exempt from `dead_code`, but a non-public fn that is only consumed by tests until a later task wires real callers may need a temporary `#[allow(dead_code)]`; remove it once production code calls it.

- **ts-rs binding generation workflow.** Wire-contract DTOs live in `src-tauri/src/` and derive `TS` with `#[ts(export, export_to = "../../src/bindings/")]`. The `export_to` path is relative to the **source-file directory** (`src-tauri/src/`), so `"../../src/bindings/"` resolves to the repo's `src/bindings/`. Bindings are regenerated whenever the Rust tests run (`cargo nextest run -p simple-archiver`). **Filtered runs do not regenerate bindings (harness â€” PR10):** a filtered invocation such as `cargo nextest run -p simple-archiver <specific_test>` omits the ts-rs export test and therefore does NOT write to `src/bindings/`. Binding drift (and the corresponding `git checkout -- src/bindings/` restore) only applies after a full `-p simple-archiver` run (or any run whose filter includes the export test). After a filtered single-test run, the bindings directory is untouched and no restore is needed. The generated `.ts` files are **committed** â€” they are a derived artifact with a single authored source (the Rust DTO) and the frontend imports them directly. A CI "bindings freshness" check (`git diff --exit-code src/bindings`) is deferred to a follow-up PR because the `src-tauri` codegen step requires a pre-built `dist/`; within each PR the contract is locked instead by Rust-side serde-shape tests. **Regen produces unformatted output (harness):** the regenerated `.ts` output is raw/unformatted, whereas committed versions are oxfmt-formatted. Running any Rust test that builds the `src-tauri` crate (workspace nextest, `cargo nextest run -p simple-archiver`, or some `cargo check -p simple-archiver` flows) leaves `src/bindings/*.ts` showing as modified with formatting-only diffs (brace style, `,`-vs-`;` terminators, indentation, trailing spaces, and occasional doc-comment rewording) â€” even when no DTO field *type* changed. If no DTO actually changed, restore with `git checkout -- src/bindings/`; if a DTO did change, re-run `pnpm format` to reformat before committing â€” otherwise the working tree looks dirty and `pnpm fmt:check` fails on the unformatted regen. **CI gap (known limitation):** the `app` CI job (`cargo nextest run -p simple-archiver`) regenerates ts-rs bindings but does NOT run `pnpm fmt:check` afterward, so binding-format drift is only caught by the `frontend` job â€” which runs on a fresh, non-regenerated checkout. This is a pre-existing latent gap; a follow-up PR should add `pnpm fmt:check` (or a `git diff --exit-code src/bindings`) to the `app` job after the nextest step.

  **DTO change workflow â€” exact steps (harness).** When a DTO field or enum variant is added or changed (e.g. `SourceKind::Zip`, `ProgressEvent.overallEtaMs`):
  1. Edit the Rust DTO in `src-tauri/src/presentation/dto.rs`.
  2. Regenerate bindings by running the named export test â€” there is NO package.json or justfile script for this; only the test triggers it:
     ```
     cargo test export_typescript_bindings --manifest-path src-tauri/Cargo.toml
     ```
  3. Run `pnpm format` (which runs oxfmt over `src/`) to normalize the raw ts-rs output to the committed format.
  4. Verify with `git diff src/bindings` that only the intended change is present â€” no spurious reformatting of unrelated bindings.
  5. If you only ran tests and the binding CONTENT is unchanged (type-identical regen), discard the formatting churn instead: `git checkout -- src/bindings/`.
  6. Run `pnpm tsc --noEmit` (or `pnpm build`) â€” Vitest does not type-check, so every test file that constructs a literal of the changed type must be backfilled before this passes.

  Skipping step 3 leaves `src/bindings/*.ts` in raw ts-rs format; `pnpm fmt:check` (the CI frontend gate) will then fail on the bindings even when no type changed.
- **HARNESS â€” ts-rs `u64` â†’ always `#[ts(type = "number")]`.** ts-rs 10 maps Rust `u64` to TypeScript `bigint`, but Tauri IPC delivers JSON `number`. A `bigint`-typed binding mismatches the runtime payload. Annotate every `u64` DTO field with `#[ts(type = "number")]` (safe for values below 2^53; byte counters in practice). `u32` maps to `number` without an override. Pair this with a regression test asserting that the generated binding file emits `number` and not `bigint`. **Extension (PR9) â€” `Option<u64>` fields need `#[ts(type = "number | null")]`.** ts-rs would emit `bigint | null` for `Option<u64>`; Tauri IPC delivers a JSON number-or-null. Override with `#[ts(type = "number | null")]` and pin it with a binding-shape test asserting the output contains `"number | null"` and does not contain `"bigint"`.
- **`unrar` on Windows MSVC needs `advapi32` linked.** `unrar_sys` bundles UnRAR C++ source that references advapi32 APIs (registry, process-token/SID, legacy CryptoAPI, `SetFileSecurityW`), but its build script never emits the link directive. Any artifact pulling in `unrar` â€” the core test binary and the Tauri app â€” fails MSVC link with `LNK1120: 13 unresolved externals` (`__imp_Reg*`, `OpenProcessToken`, `Crypt*`, â€¦). Fix: `crates/core/build.rs` emits `cargo:rustc-link-lib=advapi32` gated on `CARGO_CFG_TARGET_OS == "windows"` (inert on macOS/Linux). This defect is invisible on local macOS development and surfaces only on the CI `app` Windows runner â€” making the Windows matrix job the load-bearing gate for this dependency's link correctness.

- **App-shell composes only semantic tokens (PR11).** `AppShell` and all PR11 shell components use exclusively existing semantic tokens (`bg-background`, `bg-card`, `text-foreground`, `text-muted-foreground`, `border-border`, â€¦) plus PR10's `status-*` / `category-*` tiers â€” no new color tokens are introduced, so light/dark parity is automatic and free.
- **Native browse dialog is mode-exclusive (PR11).** `OpenDialogOptions.directory` is a boolean â€” files and folders are mutually-exclusive dialog modes. Therefore the add-source affordance is two explicit browse buttons (`AddSourceButtons`: "Add files" / "Add folder") plus drag-and-drop as the hero (D&D accepts both kinds). A single "Select" button is impossible at the OS/API level. Per WCAG, drag-and-drop must not be the sole path â€” the two explicit buttons are that fallback.
- **`token-only` test gate (PR11).** `src/__tests__/token-only.test.ts` scans source files for raw Tailwind palette utility classes (e.g. `bg-blue-100`, `text-red-500`) and fails `pnpm test` if any leak into app code. This guards the "design-system-tokens-only" acceptance criterion automatically on every `pnpm test` run.
- **`min-h-0` on the scrolling flex child is load-bearing (PR11).** The single-scroll-region layout requires `min-h-0 overflow-y-auto flex-1` on the `main` zone. Without `min-h-0`, a flex child refuses to shrink below its content height and the inner scroll silently breaks â€” the page scrolls instead of the queue.
- **Source-scanning tests must use Vite's `import.meta.glob`, not Node built-ins (PR11).** A test that scans source text (e.g. the `token-only` gate) must read files via `import.meta.glob("/src/**/*.{ts,tsx}", { query: "?raw", import: "default", eager: true })`, NOT `node:fs`/`node:path`/`process`. The frontend `tsconfig` does not include `@types/node`, so Node built-ins type-check fine under Vitest (esbuild strips types) but FAIL `pnpm build` (`tsc`), which is the real type gate. `import.meta.glob` is typed via `vite/client` (referenced in `src/vite-env.d.ts`) and needs no new dependency.
- **zustand per-row memoization: select flat scalars, never whole store objects (PR11 â€” `TaskRow`).** A `React.memo` list row that subscribes via `useJobStore(useShallow(...))` must return only flat scalars/strings from its selector (e.g. `liveBytesDone`, a precomputed `status` string), computing any derived value INSIDE the selector. Selecting whole objects (`progress`, `summary`, `perTask[index]`) defeats the memoization, because `applyProgress` rebuilds those references on every progress tick, so `useShallow`'s shallow compare sees a changed reference and re-renders EVERY row each tick. Store action references (e.g. `reorder`) are stable and safe to include.
- **Icons are `lucide-react`, used as icon + visible label (PR11 follow-up).** Hand-rolled inline SVGs (theme toggle, browse buttons, empty-queue glyph) were replaced with lucide glyphs.
